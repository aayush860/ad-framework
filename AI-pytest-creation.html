Key engineering decisions

🔁

Retry logic

Each conversion retries up to 3 times on JSON parse failure or empty response — making the pipeline resilient to LLM inconsistency.

🌡️

temperature=0

Deterministic output is critical when generating code. Temperature 0 ensures the JSON schema is always respected, not hallucinated.

📐

Strict schema rules

The system prompt explicitly handles: multi-signal, multi-system, toggle semantics, "reflects" vs "stops reflecting" — covering all edge cases in the test matrix.

Stage 3 of 3

JSON → runnable pytest file

3_pytest_generator.py reads the JSON contracts and emits a complete test_siem_e2e.py — with real function names, subscriptions, waits, and assertions.

03

Code generation from structured data

The generator iterates the JSON, interleaves validations and toggles in the correct order, and injects time.sleep() calls for eventual-consistency. No templating library needed — pure Python string building.

Runtime Layer

Adaptive validation with eventual consistency

The generated tests call two runtime helpers: apply_subscription() and validate_signal(). Both are backed by real API clients and a polling loop.

04

Polling instead of fixed sleeps

Observability systems like Datadog and TrafficPeak have eventual consistency — data may arrive seconds or minutes after a subscription change. The wait_for_condition() helper polls until the expected state is confirmed or times out.

🐍 runtime_helper.py
runtime_helper.py — wait_for_condition() 
def wait_for_condition(fn, expected,
        timeout=60, interval=15):
  start = time.time()
  while time.time() - start < timeout:
    last_value = fn()
    if last_value == expected:
      return          # ✅ condition met
    time.sleep(interval)

  raise AssertionError(
    f"Timeout: expected={expected}, got={last_value}"
  )

# Datadog logs validation
wait_for_condition(
  fn=lambda: dd.check_datadog_logs(),
  expected=expected,
  timeout=60, interval=15
)

# TrafficPeak metrics validation
wait_for_condition(
  fn=lambda: tp.check_trafficpeak_metrics(
    agent_ids=["3534db19-..."]
  ),
  expected=expected,
  timeout=60, interval=15
)
📡

Real API calls

Each validation hits the actual Datadog or TrafficPeak API — not mocks. Tests reflect real system state.

⏱️

Graceful timeouts

If a signal doesn't appear within the timeout window, a descriptive AssertionError is raised — pinpointing exactly which system and signal failed.

🔌

Lazy clients

API clients are created only when a test actually runs — no side effects at import time, making the test module safe to collect and inspect.

JSON schema at a glance

Every test case follows this exact shape — subscriptions (initial state), validations (assertions), and toggles (UI actions).

JSON test case schema 
{
  "name": string,                   // human-readable test name

  "subscriptions": [{
    "system": "datadog" | "trafficpeak",
    "signals": {
      "logs": true | false,       // initial enabled state
      "metrics": true | false    // (only signals mentioned)
    }
  }],

  "validations": [{
    "system": "datadog" | "trafficpeak",
    "signal": "logs" | "metrics",
    "expected": true | false   // reflects → true / stops → false
  }],

  "toggles": [{
    "system": "datadog" | "trafficpeak",
    "signal": "logs" | "metrics",
    "to": true | false,          // new state after UI action
    "expected": true | false
  }]
}
