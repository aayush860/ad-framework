Jenkins Job — Parameters & Triggers
Jenkins
▼
Jenkins job is triggered manually or on schedule. Parameters defined in the Jenkins UI are passed into the build. These become environment variables that flow all the way into the Python test code.
Runtime Config
contractID, connector_name, cloudzone, cleanup
Test Selection
inclusion_tags, exclusion_tags, tests_to_run
Re-run Control
Rebuild_on_Failure, Pass_Percentage_To_Re_Run
Reporting
QASE_API_TOKEN, QASE_RUN_ID, QASE_PROJECT
build_step.sh — tag processing bash 
# Jenkins params → Robot -i / -e / -t flags
if [ ! -z "$inclusion_tags" ]; then
    for tag in $inclusion_tags; do
        include_tags+="-i $tag "   # → -i smoke -i ad-directory
    done
fi

# Result: include_tags="-i smoke -i ad-directory"
#         exclude_tags="-e fips_connector"

2
Git Clone + Docker Image Build
Docker
▼
Fresh code is cloned from the specified branch. The Dockerfile from the repo is used to build a self-contained image with everything pre-installed — Python, pytest, Robot Framework, ldap3, selenium, all packages. Every run gets a clean identical environment regardless of which Jenkins agent picks up the job.
build_step.sh bash 
# 1. Clone fresh code
git clone -b $git_branch https://git.company.com/dyno.git

# 2. Copy Dockerfile from repo
sudo cp dyno/lib/dyno/products/hybrid_griffin/Dockerfile .

# 3. Build image — installs all deps inside
sudo docker build -t auto_griffin .
Dockerfile — key installs dockerfile 
FROM ubuntu:focal
# System packages
RUN apt-get install -y python3 python3-pip git ssh ldap3 ...

# Python packages
RUN pip3 install \
    pytest==7.2.2 \
    robotframework==6.1.1 \
    ldap3==2.9.1 \
    qase-robotframework==2.0.1 \
    selenium requests deepdiff ...

# Copy entire dyno project into image
COPY dyno/tests  ${DYNO_HOME}/tests
COPY dyno/lib    ${DYNO_HOME}/lib
COPY dyno/bin    ${DYNO_HOME}/bin

ENTRYPOINT ["dyno-entrypoint.sh"]
Why Docker? No dependency conflicts. Same Python version. Same package versions. Isolated from the Jenkins agent's OS. Clean state every run.

Container Starts — entrypoint.sh + Environment Injection
Docker
▼
When the container starts, entrypoint.sh runs first — before any test command. It sets up SSH agent and loads the access key needed to communicate with internal infrastructure. Only then does it hand control over to the robot command.
docker run
→
entrypoint.sh
→
ssh-agent starts
→
key loaded
→
exec robot cmd
build_step.sh — docker run bash 
sudo docker run \
  -v ${PWD}:/tmp \                    # Jenkins workspace ↔ /tmp in container
  -e contractID=$contractID \       # Jenkins param → env var in container
  -e connector_name=$connector_name \  # read by os.environ.get() in Python
  -e cleanup=$cleanup \
  -e cloudzone=$cloudzone \
  auto_griffin \
  /bin/sh -c "cd /tmp/dyno/bin && ./dyno -m robot ..."
entrypoint.sh — simplified bash 
# Start SSH agent for internal access
eval `ssh-agent -s -a /ssh-agent/agent`

# Load access key (needed to reach connectors/infra)
ssh-add /root/.ssh/akanebula.key

# Hand over to robot command
exec "$@"
Volume mount -v ${PWD}:/tmp is the bridge — test reports written inside the container at /tmp/dyno/bin/ automatically appear on the Jenkins agent workspace after the container exits.

4
Robot Framework — The Bridge to Python
Robot Framework
▼
Robot Framework runs the .robot file. The robot file contains zero test logic. It simply imports the Python pytest class as a Library, maps lifecycle hooks, and calls Python methods by name. Robot Framework acts purely as an orchestrator and reporter.
Robot .robot file
→ Library imports Python class
→ Suite Setup maps to setup_class
→ Test Setup maps to setup
→ Test Teardown maps to teardown
→ Suite Teardown maps to teardown_class
→ Each test calls Python method by name
→ Tags added for filtering
⇄
Python / pytest class
→ setup_class() — dir + IDP creation
→ setup() — per test init + AD cleanup
→ teardown() — delete groups + sync
→ teardown_class() — delete dir + IDP
→ test_01_...() actual test logic
→ All business logic + assertions
Test_Active_Directory.robot robot 
*** Settings ***
# Import Python class as Robot Library
Library    Test_Active_Directory.py  WITH NAME  obj

Suite Setup      obj.setup_class     # runs once — creates dir + IDP
Test Setup       obj.setup           # runs before each test
Test Teardown    obj.teardown        # runs after each test
Suite Teardown   obj.teardown_class  # runs once — cleanup

*** Test Cases ***

    [Documentation]  Create configure and check if directory is up
    [Tags]  smoke  ad-directory  dir  uc_ext_sanity  egt_advanced

pytest Test Execution — The Actual Tests
pytest
▼
Robot Framework calls the Python methods. This is where the actual test logic runs. Config is read from environment variables (injected by Docker). Tests interact with EAA via the API layer, AD via LDAP, and validate results.
Test_Active_Directory.py — how env vars become config python 
class Test_Active_Directory:

    @classmethod
    def setup_class(cls):
        # Jenkins params → Docker -e flag → os.environ
        cls.cleanup        = os.environ.get("cleanup", "YES")
        cls.connector_name = os.environ.get("connector_name")

        # Static config from JSON files
        cls.dirConfigData  = Config_Provider.get_param("DirectoryConfig")

        # Create directory in EAA → verify → create IDP
        cls.dir_name, cls.dir_uuid, cls.result = \
            cls.dir_generic_utils\
            .create_configure_verify_if_directory_is_up(
                dir_type="AD",
                dir_config=cls.dirConfigData['ActiveDirectory2'])

QASE — Real-time Test Management
QASE
▼
QASE is a test management tool. The QASE listener hooks into Robot Framework events and pushes results to the QASE dashboard in real time as each test completes — you don't have to wait for the run to finish to see results.
robot command — QASE integration bash 
# QASE env vars tell the listener where to push results
QASE_MODE=testops
QASE_TESTOPS_RUN_ID=${QASE_RUN_ID}
QASE_TESTOPS_PROJECT=${QASE_PROJECT}

./dyno -m robot \
  # listener hooks into Robot Framework events
  --listener qaseio.robotframework.Listener \
  Test_Active_Directory.robot

Real-time push

Run tracking

7
Smart Re-run Logic — Threshold Based
Jenkins
▼
After the first run, Jenkins reads output.xml from the volume-mounted workspace, calculates pass percentage, and decides whether to re-run failed tests.
pass% == 100
All passed. Exit 0, done. No re-run needed.
pass% > threshold
Re-run only failed tests using --rerunfailed output.xml → merge results with rebot --merge
pass% < threshold
Too many failures. Exit 1, fail the build. Not worth re-running.
Rebuild = No
Re-run disabled. Exit 1 regardless of pass percentage.
build_step.sh — rerun + merge bash 
# Re-run only failed tests
./dyno -m robot \
    --rerunfailed output.xml \    # reads failures from first run
    -o finalo.xml \               # writes re-run results separately
    Test_Active_Directory.robot

# Merge first run + re-run into one clean report
rebot --output output.xml \
      --merge output.xml finalo.xml
# Result: failed test that passed on re-run → shows as PASS

Reports — Robot HTML + Jenkins Dashboard
Reports
▼
Robot Framework automatically generates three output files. These come back to the Jenkins workspace via the volume mount. Jenkins Robot Framework Plugin reads output.xml and displays results in the Jenkins UI.
📄
output.xml
Raw machine-readable results. Read by Jenkins plugin and rebot merge.
📋
log.html
Detailed step-by-step execution log. Every keyword, every variable value.
📊
report.html
High-level pass/fail summary with statistics and tag breakdown.
Jenkins Plugin
Robot Framework Plugin reads output.xml → displays pass/fail count per build
Trend Graph
Jenkins plots pass/fail trend across all historical builds
Build Status
Build marked PASS / UNSTABLE / FAIL based on configurable thresholds
QASE Dashboard
Real-time results already pushed. Historical test run analytics available.
