Section 1
Framework Used
What technologies power the framework and why
🧪
pytest — Primary Test Framework
All test logic is written in Python using pytest conventions. Class-based tests, setup/teardown lifecycle, assert statements. This is the engine.
🤖
Robot Framework — Reporting Only
.robot files wrap the pytest classes purely for Jenkins reporting and tag-based test filtering. Zero test logic lives here.
🐳
Docker — Environment Isolation
Fresh image built every Jenkins run. Ensures clean, identical environment regardless of which Jenkins agent picks up the job.
⚙️
Jenkins — CI Orchestration
Triggers test runs. Injects parameters as environment variables. Displays Robot HTML reports and trend graphs. Webhook triggered on environment upgrades.
ToolRoleWhy this tool
pytest Test framework Industry standard Python test framework. Class-based structure, fixtures, rich assertion support.
Robot Framework Reporting + tag filtering Already established in team's Jenkins pipeline. Tags enable selective runs without code changes.
ldap3 AD/LDAP operations Directly create/modify users and groups in Active Directory from Python.
requests HTTP/API calls All EAA REST API interactions and IDP login validation.
QASE Test management Real-time result push via Robot listener. Tags map to QASE test case IDs (Q-473).
Docker Environment consistency Eliminates "works on my machine" issues. Same Python version + packages every run.

Section 2
Design Patterns Used
How the code is structured and why — the decisions behind the architecture
1
Layered Architecture
The framework is split into 4 layers — Test, Utility, API, and Data. Each layer has one responsibility. Tests don't make API calls. Utilities don't contain test assertions. Clean separation throughout.
Test → Utility → API Layer → EAA
Each layer only talks to the one below it
2
API Object Pattern
All EAA REST calls go through Directory_Apis class. Tests never make raw HTTP calls. Same concept as Page Object Model in UI testing — but applied to APIs. If API changes, only apis.py updates.
test → dir_apis.sync_directory()
never → requests.post("/api/sync")
3
Data-Driven Testing
Test data is externalized to JSON config files. Config_Provider reads them at runtime. Dynamic values like connector name and environment injected as environment variables by Jenkins.
Directory_Config.json → Config_Provider
connector_name → os.environ.get()
4
Setup / Teardown Pattern
Two-level lifecycle management. Class level for expensive shared resources (directory, IDP — created once). Method level for per-test cleanup (delete groups, sync after each test).
setup_class → create dir + IDP (once)
teardown → delete groups (per test)
5
Utility / Helper Pattern
Complex reusable workflows live in utility classes. GenericUtils holds cross-directory workflows. ADutils holds AD-specific LDAP operations. Tests call these — no logic duplication.
generic_utils.create_sync_and_validate()
ad_utils.diff_sync_operations()
6
Factory Pattern
GenerateApis reads apis.json and dynamically creates API client methods at runtime. No need to manually write HTTP boilerplate for every endpoint. Add new API = update JSON only.
GenerateApis(json_path=apis.json)
→ auto generates all client methods
7
Retry / Polling Pattern
EAA sync is async — you can't assert immediately. Polling loops with configurable timeout and poll interval wait for the expected state before asserting. Used for directory verification, user sync, login validation.
wait_till_all_user_are_synced()
directory_verification() with retries
8
What Could Be Better
generic_utils is a God Class — too many responsibilities. Test files are too large. No conftest.py — setup_class used instead of fixtures. No pytest markers — Robot tags used instead.
Anti-pattern: God Class in generic_utils
Should split by responsibility

📋 apis.json

The API specification file. GenerateApis factory reads this and dynamically creates all API client methods at runtime. Adding a new EAA endpoint only requires updating this JSON — no Python boilerplate needed. This is the Factory Pattern in action.

@classmethod
def setup_class(cls):
    # Dynamic — from Jenkins via Docker -e flag
    cls.cleanup        = os.environ.get("cleanup", "YES")
    cls.connector_name = os.environ.get("connector_name")

    # Static — from JSON config files
    cls.dirConfigData  = Config_Provider.get_param("DirectoryConfig")
    cls.attrConfigData = Config_Provider.get_param("CustomAttrData")
Smart re-run logic
ConditionActionWhy
pass% == 100 Exit 0 — done All tests passed. No re-run needed.
pass% > threshold Re-run failed only → merge Likely flaky tests. Re-run just failures, merge with rebot.
pass% < threshold Exit 1 — fail build Too many failures. Real issue. Not worth re-running.
Rebuild_on_Failure = No Exit 1 — no re-run Re-run disabled by Jenkins parameter.
