Design Patterns Applied

Framework Architecture Patterns

Patterns used across the adapter, infrastructure, and configuration layers

01
Adapter Pattern
Locust is a generic load engine — it has no idea what a log or metric payload looks like. The adapter sits in the middle and translates Locust's task trigger into exactly what the target API expects.
Locust Engine
knows RPSspawns usersfires @task
Adapter Layer
modify_payload()attach certset headerspick protocol
Target API
Akamai endpointPOST /logs → JSONPOST /telegraf → InfluxDB
02
Adapter File Structure
Every adapter file follows the same 3-part structure. Add a new platform? Create a new file with the same shape — nothing else changes.
📄logs_locust_datadog.py
Part 1 — Config Provider
class Config: reads tenant file, template, RPS, cert paths CONFIG = Config() # singleton — created once at startup
↓
Part 2 — Base Locust Class (Template Method)
class BaseLogsUser(HttpUser): tenant = None # ← variable slot, filled by factory def on_start(): # load cert + headers — runs once per instance @task def send_log(): # modify payload → POST request
↓
Part 3 — Factory Class Creator
for tenant in TENANTS: type(class_name, (BaseLogsUser,), {"tenant": tenant}) # stamps 1 unique class per tenant at startup # all sit in memory until the test stops
03
Template Method Pattern
The base Locust class defines the fixed algorithm — how to send. The tenant is the only variable — who is sending. Each generated subclass inherits the full algorithm but plugs in a different tenant identity.
Template Method ConceptYour Code
Abstract base classBaseLogsUser / BaseMetricsUser
Fixed algorithm skeletonon_start() + @task send_log()
Variable / hook parttenant = None — overridden per class
Concrete subclassesDynamically generated per tenant at startup
04
Factory Pattern
100 tenants, each with different certs and config. Writing one class manually per tenant is not an option. The factory reads the tenant list at startup and manufactures all classes in one go.
⚡
Cert reloading is expensive
Each tenant has a unique .crt + .key. Reloading on every request at 100 RPS is wasteful. on_start() loads it once per instance and reuses it.
⚖️
Locust scales per user class
With 1 class for all tenants you lose per-tenant RPS control. 1 class per tenant = Locust distributes workers independently per tenant.
🏭
type() manufactures classes at runtime
100 tenants in list_of_tenants.txt → 100 unique Locust classes stamped at startup → all sit in memory until the test stops.
05
Master-Worker Architecture
At the infrastructure level, Locust runs in a distributed master-worker model on Kubernetes. One master coordinates and reports; N workers generate concurrent load independently.
👑
Master Pod
Hosts Locust UI · coordinates workers · aggregates results
⚙️
Worker 1
runs tenant classes · fires requests
⚙️
Worker 2
runs tenant classes · fires requests
⚙️
Worker N
runs tenant classes · fires requests
4 separate master-worker clusters run in parallel — DD Logs, DD Metrics, TP Logs, TP Metrics — each scaling independently.

🔀
Separation of Concerns
Each folder has exactly one job. Adapters only know about load generation. Utility only handles pre-setup. Data only holds static inputs. Nothing bleeds into each other.
🔓
Open / Closed Principle
Adding a new platform means adding one folder in each of the 4 places. No existing files are touched. Open for extension, closed for modification.
🧭
Discoverability
Any developer can navigate to exactly what they need without reading code. Datadog logs config? → data/access_logs/datadog/. TrafficPeak K8s? → k8s/trafficpeak-logs/.
🔁
Consistency
Every platform follows the identical folder naming convention — logs vs metrics, then platform name. Same pattern repeated across all 4 top-level folders.
